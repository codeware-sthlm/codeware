# Design System

This Storybook covers the UI and theming layer for the Codeware platform. Two concerns live here:

- **Dashboard components** — `libs/app-cms/ui/dashboard`. Purpose-built components for Payload CMS admin customizations: cards, banners, modals, buttons, form inputs.
- **Theme system** — `libs/shared/theme`. CSS custom property sets that drive visual identity across the CMS admin and tenant web apps, with Tailwind v4 wiring.

More libraries will appear as they are added to `apps/storybook/.storybook/main.ts`.

---

## Toolbar controls

Two independent controls sit in the Storybook toolbar:

| Control         | Attribute set   | Effect                                              |
| --------------- | --------------- | --------------------------------------------------- |
| **App Theme**   | `data-sb-theme` | Activates a named token set — palette, radius, etc. |
| **Color Theme** | `data-theme`    | Toggles `light` / `dark` within the active theme    |

Every story is wrapped by the decorator in `preview.tsx`:

```html
<div
  data-sb-theme="payload-admin"
  data-theme="light"
  class="twp bg-background p-6"
>
  <!-- story renders here -->
</div>
```

Switching the toolbar changes these attributes at runtime. Because tokens resolve through CSS custom properties, the visual output updates instantly — no rebuild.

## Token system

Every theme is wired into Tailwind through three layers:

<pre className="mermaid">
  {`flowchart TD
  A["1  Theme values\\ntheme-light.css / theme-dark.css\\n--background, --primary, --brand-500"]
  B["2  Tailwind bridge\\n_core/tailwind-setup.css  @theme inline\\n--color-background, --color-primary, --color-brand-500"]
  C["3  Utility classes\\nbg-background, bg-primary, bg-brand-500"]

  A -->|"CSS cascade"| B
  B -->|"runtime resolve"| C`}
</pre>

**Layer 1 — Theme values** (`theme-light.css` / `theme-dark.css`)

Semantic CSS variables defined per theme and per color mode:

```css
--background: oklch(1 0 0);
--primary: oklch(0.21 0.006 285.885);
--brand-500: var(--color-teal-500);
/* …full set of shadcn tokens + brand palette entries… */
```

**Layer 2 — Tailwind bridge** (`_core/tailwind-setup.css`)

`@theme inline` maps theme variables to Tailwind's `--color-*` naming convention. The `inline` modifier means values resolve at runtime, not build time:

```css
@theme inline {
  --color-background: var(--background);
  --color-primary: var(--primary);
  --color-brand-500: var(--brand-500);
  /* …all mappings… */
}
```

**Layer 3 — Utility classes**

Tailwind generates color utilities from every `--color-*` variable:

| CSS variable         | Generated utilities                          |
| -------------------- | -------------------------------------------- |
| `--color-background` | `bg-background`, `text-background`           |
| `--color-primary`    | `bg-primary`, `text-primary`, `ring-primary` |
| `--color-brand-500`  | `bg-brand-500`, `text-brand-500`             |

Switching `data-sb-theme` swaps the variable values — all `bg-*`, `text-*`, and `border-*` utilities immediately reflect the new theme without a rebuild.

### Dark mode

Each theme's `tailwind-base.css` imports `_core/tailwind-setup.css`, which declares `@custom-variant dark` using the `[data-theme=dark]` attribute strategy. The Storybook toolbar maps to this via the decorator:

<pre className="mermaid">
  {`flowchart TD
  toolbar["Color Theme toolbar"]

  toolbar -->|"payload-admin"| attr["data-theme=dark on wrapper\\nmatches @custom-variant"]
  toolbar -->|"shadcn / spotlight / codeware"| cls[".dark class on wrapper\\nthemes.css scoped selectors"]

  attr --> active["dark: utilities activate"]
  cls --> active`}
</pre>

`payload-admin` uses the attribute strategy because Payload CMS sets `data-theme` on `<html>` — not a CSS class. The other themes use `.dark` class, matched by scoped selectors in the auto-generated `themes.css`. The decorator derives the correct strategy from `THEME_DARK_STRATEGIES` in `themes-meta.ts`.

### Brand tokens

`--brand-*` is the semantic brand color layer — each theme maps it to a concrete palette:

| Theme           | `--brand-500` resolves to |
| --------------- | ------------------------- |
| `shadcn`        | `var(--color-zinc-500)`   |
| `payload-admin` | `var(--color-teal-500)`   |
| `spotlight`     | `var(--color-teal-500)`   |
| `codeware`      | `var(--color-blue-500)`   |

Dashboard components use `brand-*` utilities (`bg-brand-500`, `text-brand-600`, etc.) rather than hardcoded color names — the same component renders teal in Payload Admin and blue in Codeware without any code changes. `shadcn` maps `--brand-*` to zinc to satisfy the token contract without asserting a brand identity.

## Relation to shadcn

[shadcn/ui](https://ui.shadcn.com) is a component collection distributed as copy-paste source (not an npm package). Each component uses CSS custom properties for color — `--background`, `--foreground`, `--primary`, `--muted`, etc. — with a defined naming convention.

Our `theme-light.css` / `theme-dark.css` files implement exactly those same variable names. This means:

- Any shadcn component copied into `libs/shared/ui/` inherits the active theme's colors without extra configuration.
- Adding a new shadcn component that uses a variable you haven't defined in a theme causes that value to fall through to nothing (transparent or inherited). Check the [shadcn variable list](https://ui.shadcn.com/docs/theming#list-of-variables) when adding new components.
- The dashboard components in this Storybook are _not_ shadcn — they are custom-built. But they use the same variables so they theme correctly alongside shadcn primitives.
- shadcn v2 targets Tailwind v4 with CSS-first config. This repo uses Tailwind v4 (`@import 'tailwindcss'`), so shadcn components work as documented without adapter shims.

## Keeping shadcn components up to date

shadcn distributes components as source — there is no npm package to bump. Updates are applied by re-downloading the latest component templates via the CLI.

### Update all components

```sh
pnpm nx shadcn:update
```

This runs three steps in sequence:

1. `shadcn apply --preset <code>` — re-downloads all installed components from the nova registry using the preset derived from `components.json`
2. `nx format` — normalises Prettier formatting (shadcn writes its own style)
3. `nx run-many --target=lint --fix` — fixes ESLint import ordering

After running, review the diff and verify stories in Storybook before committing.

### Add a new component

```sh
pnpm nx shadcn:add -- <component>
```

> **Note:** the nova-style registry (`/r/styles/nova/<component>.json`) currently returns 404 for individual `add` requests. If this fails, add the component via a new preset on [ui.shadcn.com/create](https://ui.shadcn.com/create) and re-run `shadcn:update`.

### Detect drift

`shadcn add <component> --diff` does not work with the nova style (same 404 issue). To check whether upstream has changed a component:

1. Run `pnpm nx shadcn:update` on a clean branch
2. Review `git diff` — changed files indicate upstream drift
3. Discard or keep changes as appropriate

### Check current preset

```sh
pnpm nx shadcn:preset
```

Prints the resolved preset code and all installed components. The preset code comes from `components.json` settings; it changes only if you modify the style, base colour, or icon library.

## Keeping `themes.css` in sync

  {`flowchart TD

    end
  end

    css_out["themes.css"]
    ts_out["themes-meta.ts"]
    prev_css["preview.css"]

  end

  light_src --> gen

  setup_src --> gen
  prose_src --> gen
  gen --> css_out
  gen --> ts_out
  css_out --> prev_css
  ts_out --> prev_tsx`}
</pre>

`themes.css` and `themes-meta.ts` are auto-generated — never edit them by hand. Two files in `_core/` act as the **contract blueprint** against which every theme is validated:

| Blueprint file              | Defines                                                                     |
| --------------------------- | --------------------------------------------------------------------------- |
| `_core/tailwind-setup.css`  | All CSS variables that must be present in every theme (via `@theme inline`) |
| `_core/typography-prose.js` | All prose CSS variables required for typography styling                     |

The generator reads each theme's token files, checks completeness against both blueprints, detects the dark strategy from the opening selector of `theme-dark.css`, and writes the scoped CSS variables and TypeScript exports consumed by the Storybook preview. A theme missing any blueprint token causes sync to fail with a completeness error.

> **Hard requirement.** Nx enforces that these files stay in sync with their source tokens. `pnpm nx sync` will fail during CI if the generated files are out of date, and the workspace will be marked out of sync locally until they are regenerated. Any change to a theme's token files or to `tailwind-setup.css` must be followed by a sync before committing.

```sh
pnpm nx sync     # sync files when needed (run after any token change)
```

After modifying the generator itself, rebuild the plugin with `pnpm nx build dev-plugin` before running sync.

To verify a theme visually after syncing, open **Theme › Overview** in the sidebar — it renders every token as a color swatch for all four themes side by side, with light and dark variants.

## Adding a new theme

**1. Create the theme directory**

```
libs/shared/theme/src/lib/{your-theme}/
```

**2. Add `theme-light.css`**

Copy `payload-admin/theme-light.css` as a starting point. Set all shadcn tokens and `--brand-*` palette entries:

```css
:root {
  --radius: 0.5rem;
  --background: oklch(1 0 0);
  --foreground: oklch(0.14 0.005 285);
  /* ... all shadcn tokens ... */

  --brand-50: var(--color-indigo-50);
  --brand-500: var(--color-indigo-500);
  --brand-950: var(--color-indigo-950);
  /* ... full 50–950 scale ... */
}
```

**3. Add `theme-dark.css`**

Only override the tokens that change in dark mode. The **opening selector** is read by the sync generator to auto-detect the dark strategy:

- `.dark { … }` → class strategy (decorator adds `.dark` class to wrapper)
- Anything else (e.g. `:root`, `[data-theme='dark']`) → attribute strategy

**4. Add `tailwind-base.css`**

```css
@import './theme-light.css';
@import './theme-dark.css';
@import '../_core/tailwind-setup.css';
```

**5. Register in the sync generator**

Add your theme name to `STORYBOOK_THEMES` in `tools/dev-plugin/src/generators/theme-sync/theme-sync.ts`:

```ts
const STORYBOOK_THEMES = [
  'shadcn',
  'payload-admin',
  'spotlight',
  'codeware',
  'your-theme'
] as const;
```

**6. Register in the Storybook toolbar**

Add an item to `sbTheme.toolbar.items` in `apps/storybook/.storybook/preview.tsx`:

```ts
{ value: 'your-theme', title: 'Your Theme' }
```

The dark strategy is **auto-detected** from the opening selector of `theme-dark.css`. After running sync, `themes-meta.ts` is regenerated with the correct `THEME_DARK_STRATEGIES` entry — no manual edits needed in `preview.tsx`.

**7. Run sync**

```sh
pnpm nx sync
```

See [Keeping `themes.css` in sync](#keeping-themescss-in-sync) above for what the generator produces and how CI drift detection works.

**8. Use the theme in an app**

At the top of the app's CSS entry point:

```css
@import 'tailwindcss';
@import '../../../libs/shared/theme/src/lib/{your-theme}/tailwind-base.css';
```

## Using components inside Payload Admin

Payload Admin ships with its own global CSS and a browser-level reset. Applying Tailwind's preflight globally would break the admin UI, so it is scoped with the `.twp` class.

`apps/cms/src/app/(payload)/custom.css` wires the theme into Payload's admin with a single import:

```css
@import 'libs/shared/theme/src/lib/payload-admin/tailwind-base.css';
```

This brings in token definitions, the dark variant, the `@theme inline` mappings, and the base layer in one line.

### The `.twp` wrapper

<pre className="mermaid">
  {`flowchart TD
  html_node["html\\ndata-theme = light OR dark\\nPayload ThemeProvider manages this"]

  subgraph page["Payload Admin page"]
    nav_node[".nav\\nPayload sidebar\\nfont-size: 13px"]

    subgraph twp_node[".twp — Tailwind preflight scoped here"]
      comp_node["Dashboard components\\nbg-brand-500, text-foreground\\nbrand utilities work correctly"]
      notwp_node[".no-twp\\nPayload native styles restored"]
    end
  end

  portal_node["div role=dialog\\nPortaled outside .twp by Radix UI\\npreflight applied explicitly in custom.css"]

  html_node --> page
  html_node -.->|"DOM teleport"| portal_node`}
</pre>

Wrap any Tailwind-based component in `className="twp"` to activate scoped preflight:

```tsx
export function MyAdminWidget() {
  return (
    <div className="twp">
      <BrandBtn>Save</BrandBtn>
    </div>
  );
}
```

`custom.css` applies `tailwindcss/preflight.css` only inside `.twp`, resetting browser defaults for that subtree without touching Payload's own elements.

Without `.twp`, Tailwind color utilities still apply but browser defaults are not reset — buttons, inputs, and headings may pick up Payload's global styles instead of yours.

### The `.no-twp` escape hatch

Inside a `.twp` block, wrap Payload-native elements with `.no-twp` to revert to Payload's own styles:

```tsx
<div className="twp">
  <MyDashboardCard />
  <div className="no-twp">
    {/* Payload's native checkbox, select, and input styles apply here */}
  </div>
</div>
```

### Dialog and portal components

shadcn `Dialog` (and other portal-based components) teleport their content outside the `.twp` node in the DOM tree, so the scoped preflight does not apply. `custom.css` handles this with an explicit selector alongside `.twp`:

```css
.twp,
div[role='dialog'][data-slot='dialog-content'] {
  @import 'tailwindcss/preflight.css';
}
```

If you add other portal components that need scoped preflight, extend that selector list.

### Dark mode in Payload Admin

No action required.

Payload's `ThemeProvider` sets `data-theme="light"` or `data-theme="dark"` on `<html>` automatically, driven by the user's preference cookie or OS setting. Because `payload-admin`'s `@custom-variant dark` targets `[data-theme=dark]` on any ancestor, dashboard components respond to Payload's toggle without any wiring on your part.
